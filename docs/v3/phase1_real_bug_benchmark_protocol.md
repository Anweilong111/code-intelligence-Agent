# V3 Phase 1: Real Python Bug Benchmark Protocol

## Purpose

Phase 1 replaces deterministic repair fixtures with a fixed-SHA benchmark whose
failures are observed in real project tests. It establishes the input and oracle
for later live-LLM repair evaluation; it does not claim a live-model repair rate.

The accepted catalog contains 20 reproducible bugs from five projects. Five
additional cases remain visible as rejected evidence instead of being removed
from the denominator silently.

| Split | Accepted projects | Accepted cases | Rejected cases |
| --- | --- | ---: | ---: |
| development | Black, PySnooper | 7 | 1 |
| validation | FastAPI, tqdm | 8 | 0 |
| test | youtube-dl | 5 | 4 Tornado cases |
| total | 5 | 20 | 5 |

Repository families do not cross splits. This prevents a project-specific patch
pattern learned during development from appearing again in the blind test split.

## Acceptance Gates

A candidate becomes `accepted` only when one ignored raw reproduction artifact
passes every gate below:

1. The exact bug commit and fix commit are downloaded into fresh checkout slots.
2. The exact patch-level Python runtime matches the case metadata.
3. Test files listed by the benchmark, plus explicitly justified test-only helper
   files, are copied from the fix commit to the bug commit.
4. At least one targeted test executes at the bug commit and fails with a test
   assertion/error signal rather than an environment blocker.
5. The same targeted test executes at the fix commit and passes with a non-zero
   test count.
6. The fixed revision regression command passes with a non-zero test count.
7. The artifact reports no blocker and confirms that the gold patch was not
   visible to execution.
8. Case ID, bug/fix SHA, Python version, overlay paths, and command arguments
   match the current catalog exactly.

`v3_real_bug_evidence.py` validates these invariants before changing a case from
`candidate` to `accepted`. The committed catalog stores only test counts,
statuses, exact runtime version, relative artifact reference, and artifact
SHA-256. Raw stdout/stderr, local paths, checkouts, and runtime directories remain
under ignored `outputs_v3/` paths.

## Source And Ground Truth Isolation

- Source benchmark: BugsInPy at commit
  `11c5f1eea954a42132cfd06bf257766a7963e0fd`.
- Every case records the repository URL, bug SHA, fix SHA, issue or pull request,
  license, target tests, changed source files/functions, and provenance paths.
- `bug_patch.txt` is parsed as benchmark metadata to produce ground-truth labels.
  Its contents, fix-commit source, and test answers are not placed in model
  context.
- Upstream `setup.sh`, `setup.py`, Make targets, and shell commands are inspected
  as data and are not executed by the Phase 1 importer.
- Test execution accepts only normalized `python -m pytest`, `unittest`, or
  `nose` commands. Free-form shell control tokens are rejected.

## Historical Repository Adaptations

The benchmark preserves historical behavior while avoiding unbounded setup
execution. Every adaptation is represented in catalog provenance.

### Black

Historical Black imports `_black_version.py`, which was generated by packaging.
The benchmark writes a deterministic import-only version value without running
`setup.py`; it does not contain repair logic or a gold answer.

Black case 1 is rejected. Its bug target fails and fix target passes in isolation,
but the fixed full suite leaves a closed asyncio event loop before the same test.
The observed result is 127 passed, 1 failed, and 1 skipped, so the full-regression
gate is not satisfied.

### PySnooper

PySnooper case 2's fixed targeted test imports three files from the test-only
`tests/mini_toolbox` package that do not exist at the bug SHA. Those files are
explicit overlays with reasons and checksums. A fresh-checkout rerun exposed this
dependency; the earlier contaminated checkout result was discarded.

### FastAPI

Four historical dependency profiles preserve the corresponding Starlette and
Pydantic versions. Cases 4, 5, and 7 deselect one unrelated Windows-only timing
test that assumes two immediate `time.time()` calls cannot be equal. The exact
node and reason are recorded, and catalog validation rejects any exclusion that
overlaps a targeted test or patch ground-truth test function.

### tqdm

The historical functional suite runs through `python -m nose`; `tests_perf.py`
remains excluded as in upstream tox configuration. Two unrelated Windows-only
tests are explicitly excluded: ANSI terminal width handling and a legacy `eval`
of a path containing a backslash-U sequence. The target nodes are not excluded.
The required `ls` executable is resolved from an allowlisted local tool path.

### youtube-dl

The upstream Make `offlinetest` intent is represented as explicit unittest
modules. Online/download/signature and external-tool files are excluded according
to the documented offline boundary. Older fixed SHAs use only modules present at
that revision; missing future test modules are not treated as repository defects.

### Tornado Rejections

Four Tornado candidates are retained as rejected because the historical fixed
revision does not provide a clean full-regression oracle under the pinned Windows
runtime. Case 9 is the directly retained representative run: 1,258 passed, 38
failed, 38 errored, and 85 skipped. This representative evidence is not presented
as four independent successful reproductions.

## Environment Profiles

Eight profiles bind all 25 catalog entries to exact CPython patch versions and
installed package versions. FastAPI cases use separate profiles because their
historical dependency combinations differ. The generated manifest records:

- Python version, implementation, platform, and architecture;
- normalized package names and exact versions from `python -m pip list`;
- profile-to-case bindings;
- per-profile and whole-manifest SHA-256 fingerprints;
- an explicit statement that runtime directories are not committed.

See `phase1_environment_profiles.json` for the package-level snapshot and
`environment_profile_sources.json` for repository-relative runtime locators.

## Reproduction Commands

Build the candidate catalog without executing repository scripts:

```powershell
python -m code_intelligence_agent.evaluation.v3_real_bug_benchmark `
  outputs_v3/catalog_sources/BugsInPy `
  datasets/v3_real_bugs/bugsinpy_selection.json `
  docs/v3/phase1_candidate_catalog `
  --require-pass
```

Reproduce one case using its pinned historical runtime:

```powershell
python -m code_intelligence_agent.evaluation.v3_real_bug_reproduction `
  docs/v3/phase1_candidate_catalog.json `
  bugsinpy-youtube-dl-8 `
  outputs_v3/reproduction/bugsinpy-youtube-dl-8 `
  --python-executable outputs_v3/runtimes/cpython-3.7.0/python.exe `
  --targeted-timeout 180 `
  --regression-timeout 900 `
  --require-pass
```

Validate all raw evidence and produce the accepted catalog:

```powershell
python -m code_intelligence_agent.evaluation.v3_real_bug_evidence `
  docs/v3/phase1_candidate_catalog.json `
  outputs_v3/reproduction `
  docs/v3/phase1_real_bug_catalog `
  --require-complete `
  --require-pass
```

Capture environment snapshots:

```powershell
python -m code_intelligence_agent.evaluation.v3_environment_profiles `
  datasets/v3_real_bugs/environment_profile_sources.json `
  docs/v3/phase1_real_bug_catalog.json `
  docs/v3/phase1_environment_profiles `
  --root . `
  --require-pass
```

## Phase 1 Result And Limits

- Candidate catalog: 25 cases, 6 repositories, 0 import errors.
- Accepted catalog: 20 reproducible cases, 5 explicit rejections, 0 unresolved.
- Environment profiles: 8 profiles, 25/25 bindings.
- Required difficulty tags are covered: static-negative, cross-function,
  data-flow, separated failure site, high-similarity candidates, and multi-file.
- The result proves the benchmark oracle and execution chain, not Rule/LLM/Hybrid
  repair performance. Live-model trials, pass@1/pass@3, cost, latency, reflection
  recovery, and verified repair rates belong to Phase 3.
- The current benchmark is Python- and Windows-scoped and contains only five
  accepted project families. Broader repository generalization remains a later
  evaluation concern.
